## 今日要点

1. **多 Agent 的操作面正在下沉到本地会话层。**AgentFactory 选择 tmux 承载进程、SQLite 保存状态，而不是把调度状态交给常驻服务；这使终端断开或 CLI 升级不必然中断会话。
2. **Agent 评估开始从单次调用走向整段执行轨迹。**Agent Scratchpad 的规则检查重复调用、错误重试、工具集外调用和缺少最终答案等可复现模式；它刻意不把这种规则检查包装成通用“幻觉检测”。
3. **当 Agent 接触现实世界数据时，结论边界需要被编码。**Causal DAG Explorer MCP 将外部观察作为显式输入，拒绝自行查询 Home Assistant 或填补缺失数据，并把图边定位为可编辑的探索假设。
4. **计算机操作的“更少轮次”不能替代权限边界。**Accio 将界面动作与普通 Python 合并，目标是减少模型往返；其 README 同时明确这是能够访问当前用户资源的高权限本地执行环境。

## 开源项目

### 1. AgentFactory：以 tmux 承载本地 Agent 会话的统一操作面

![AgentFactory 官方 README 展示的多 Agent 终端会话仪表板](https://raw.githubusercontent.com/compuficial/agentfactory/main/docs/assets/dashboard.png)

*图：AgentFactory README 中的本地仪表板截图；原始来源：[docs/assets/dashboard.png](https://github.com/compuficial/agentfactory/blob/main/docs/assets/dashboard.png)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-24 00:18 UTC**。
- **项目：**[compuficial/agentfactory](https://github.com/compuficial/agentfactory)（AGPL-3.0）。
- **项目自述：**`af` 是一个面向 Claude Code、Codex、Grok、OpenCode 与自定义命令的本地会话管理器。README 列出 `working`、`awaiting-input`、`done` 等状态，以及 `status --json`、`send`、`peek`、`wait` 和基于 stdio 的 MCP 接口；会话运行在专用 tmux socket 中，状态存于 SQLite，工具本身不直接调用模型服务。
- **为什么值得关注：**编排多个 coding agent 时，最先失去可见性的往往不是提示词而是终端状态：谁在等待权限、谁已经完成、谁的输出需要人工接手。将这些状态提供为 CLI、TUI 和结构化 JSON，可让人或上层协调器把“轮询窗口”改为明确的等待条件。它仍是早期项目，README 所称的兼容性与生命周期行为应在目标操作系统、tmux 版本和所用 harness 上自行验证；AGPL-3.0 的网络部署与再分发义务也应在采用前审阅。

### 2. Agent Scratchpad：回放 ReAct 轨迹并定位可验证的循环故障

![Agent Scratchpad 官方 README 展示的 ReAct 轨迹回放与健康报告界面](https://raw.githubusercontent.com/kbipul/agent-scratchpad/main/docs/demo.png)

*图：Agent Scratchpad 仓库提交的演示截图；原始来源：[docs/demo.png](https://github.com/kbipul/agent-scratchpad/blob/main/docs/demo.png)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-24 00:45 UTC**。
- **项目：**[kbipul/agent-scratchpad](https://github.com/kbipul/agent-scratchpad)（MIT）。
- **项目自述：**该浏览器端工具解析带有 `Thought`、`Action`、`Action Input`、`Observation` 与 `Final Answer` 标签的 ReAct 文本，逐步回放并运行八类确定性检查。README 列举的检查包括工具集外调用、相同工具/输入的重复、A-B-A-B 振荡、原样错误重试、步数超限与缺少最终答案；项目还将“无根据答案”限制为能在最终文本中识别、却没有出现于任何 observation 的数字或引号片段。
- **为什么值得关注：**工具调用的 JSON 合法，并不代表任务循环健康。把比较范围扩展到整个轨迹、并对 JSON 键序和空白做归一化，能让回归测试捕捉到“格式略变但实质重复”的失败。与此同时，作者没有将规则引擎声称为语义级真伪判断：它只能报告规则所覆盖的、可检查的未落地细节。这种可证范围小于“检测幻觉”，但更适合作为调试与 CI 信号。

### 3. Causal DAG Explorer MCP：让证据输入与因果假设保持可分离

![Causal DAG Explorer MCP 官方 README 展示的本地 DAG 编辑器](https://raw.githubusercontent.com/resace3/causal-dag-explorer-mcp/main/docs/app-screenshot.jpg)

*图：Causal DAG Explorer MCP README 中的本地界面截图；原始来源：[docs/app-screenshot.jpg](https://github.com/resace3/causal-dag-explorer-mcp/blob/main/docs/app-screenshot.jpg)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-24 00:01 UTC**。
- **项目：**[resace3/causal-dag-explorer-mcp](https://github.com/resace3/causal-dag-explorer-mcp)（MIT）。
- **项目自述：**该项目提供本地 MCP server 与 React Flow 编辑器，用于建立、校验和保存有向无环图（DAG）。其工具可创建、读取、更新和删除本地的“day”图；输入会校验端点、重复边和有向环。`create_dag_from_ha_evidence` 只接受 MCP host 已收集并压缩的 Home Assistant 证据摘要，运行时数据保存在本地 JSON 文件，并以临时文件加原子替换写入。
- **为什么值得关注：**“将家庭设备数据交给 Agent 分析”很容易把来源、观察与推断混为一谈。该项目把数据获取留在外部 host，把提交给图的内容限制为显式提供的摘要，并明确不自行连接 Home Assistant、不保存凭据、不做因果推断或补全缺失序列。图边因而是可审查、可编辑的工作假设，而非系统对因果关系的证明；这是把 Agent 输出纳入人工复核的一种具体接口设计。

### 4. Accio Computer Use：用 Python 组合 macOS 桌面操作与普通工具调用

- **首次公开：**GitHub 仓库创建于 **2026-07-24 08:00 UTC**；该条目为本次对既有日报的增量核验发现。
- **项目：**[Accio-Work-Lab/accio-cu](https://github.com/Accio-Work-Lab/accio-cu)（Apache-2.0）。
- **项目自述：**Accio 将 GUI 操作、命令行、API 与文件处理组合在同一段 Python 中；界面动作会返回刷新后的辅助功能状态与截图工件。README 给出基于 stdio 的 MCP 配置，以及 macOS 14+、Swift 6、Python 3.10+、辅助功能与屏幕录制授权等前提。作者在 README 中公布了其 MacOSWorld 运行记录，但由于使用的模型版本不同，项目也将其表述为端到端比较，而非受控消融实验。
- **为什么值得关注：**把条件、循环、解析和局部恢复留在一个可保存的工作流内，能够减少“每一次点击都回到模型规划”的往返，并使成功步骤可以复用。安全代价同样应被视为接口的一部分：项目明确 coding 模式以当前用户权限执行普通 Python，可能访问文件、环境、网络和进程；它建议将不受信任的 Agent 放入独立账户或 VM，且不应将 daemon 或 MCP 传输暴露到网络。

## 趋势 / 编辑简评

今天的四个项目围绕的不是“让 Agent 再多做一步”，而是把执行中的不确定性放回可观察、可反驳的表面：会话是否正在等待输入，能够由本地状态看到；一段工具调用是否陷入循环，能够由轨迹规则检查；一条图边是否只是研究假设，能够由证据边界和人工编辑保留；桌面动作的便利是否越过权限边界，也应由部署模型明确回答。它们都还不能替代权限控制、端到端评估或领域专家判断，但共同提示了一个工程方向：**把 Agent 的状态、依据和失败模式设计成可读取的工件，而不是只留在对话窗口里。**
