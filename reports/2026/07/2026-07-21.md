## 今日要点

1. **Agent 的“协作层”正在产品化。**Co-Agent 将多模型咨询明确为有证据约束的议事流程；BossConsole 则把多个 CLI agent 放进统一的操作环境。
2. **可复现不只属于评测。**pi-textbook 用固定 checkpoint、测试和故障实验把 agent loop 的学习过程变成可检验的 Git 历史，而不是一次性教程。
3. **自主性需与边界一起交付。**今天的项目都把控制面放在显眼位置：证据包、可见异议、每工具开关、只读或可追溯的练习路径。它们的能力与安全声明仍应在实际部署中独立测试。

## 开源项目

### 1. pi-textbook：把 Pi-style coding agent 拆成 15 个可运行的学习断点

![动手学 Pi 的官方在线教材首页](https://raw.githubusercontent.com/hahhforest/pi-textbook/main/docs/assets/homepage.jpg)

*图：`pi-textbook` 官方 README 展示的在线教材首页。原始来源：[hahhforest/pi-textbook](https://github.com/hahhforest/pi-textbook)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-21 04:17 UTC**。
- **项目：**[hahhforest/pi-textbook](https://github.com/hahhforest/pi-textbook)（仓库标注 MIT；README 另说明教材正文与原创媒体采用 CC BY 4.0）。
- **项目自述：**从一条离线 agent 轨迹出发，依次覆盖 TypeScript 协议、流式模型、provider、工具、agent loop、会话树、上下文压缩、扩展与评测。每章配套教材、真实 commit、聚焦测试与故障实验；课程分支以 15 个 checkpoint 组织。
- **为什么值得关注：**很多 agent 教程把控制流、状态和工具调用揉进成品代码，读者难以判断某个设计为何出现。以可 checkout 的历史和可运行测试串联学习材料，能把“理解了”变成更接近可复现的验证。它是社区非官方课程，不代表 Pi 或 Earendil Works。

### 2. BossConsole：将 Agent 的工具面与治理面放到一个开源桌面工作区

![BossConsole 官方 BOSS 标识](https://raw.githubusercontent.com/risa-labs-inc/BossConsole/main/docs/boss-logo.png)

*图：BossConsole 官方 README 中的 BOSS 标识。原始来源：[risa-labs-inc/BossConsole](https://github.com/risa-labs-inc/BossConsole)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-21 01:01 UTC**。
- **项目：**[risa-labs-inc/BossConsole](https://github.com/risa-labs-inc/BossConsole)（Apache-2.0）。
- **项目自述：**基于 Kotlin/Compose Multiplatform 的桌面工作区，聚合嵌入式浏览器、终端、编辑器、插件工具箱与 MCP 工具层；README 列出 Claude Code、Codex、Gemini、OpenCode 的接入方式，并称可提供 100 余项 MCP 工具。文档还描述了角色权限、逐工具停用开关、用户范围的 secrets 与签名插件等机制。
- **为什么值得关注：**Agent 的真实风险通常出现在“它能碰到什么”，而非聊天框中。把工具发现、权限与紧急停用纳入同一个操作台，提供了一种以控制面为中心的设计方向。不过 README 中关于工具数量、隔离与安全边界的陈述均属项目方说明；在接入凭据或生产环境前，仍需检查权限配置、插件供应链和实际网络边界。

### 3. Co-Agent：让多 Agent 的分歧成为一等输出

![Co-Agent 官方 README 中的 council 演示界面](https://raw.githubusercontent.com/yuunnn/co-agent/main/assets/co-agent-council-wide.png)

*图：Co-Agent 官方 README 的 council 演示界面。原始来源：[yuunnn/co-agent](https://github.com/yuunnn/co-agent)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-21 10:13 UTC**。
- **项目：**[yuunnn/co-agent](https://github.com/yuunnn/co-agent)（Apache-2.0）。
- **项目自述：**以 Codex 当前任务为主持者，向已配置的终端 agent 或兼容 API 模型发出独立判断；随后进行对最强分歧的挑战，并产出带有支持证据、未解决争议与下一步的 Co-Result。README 描述了 prompt、workspace、sealed 三种证据模式，并声明默认将会话与证据包保存在本地。
- **为什么值得关注：**“多 agent”常退化为多个彼此不知道对方结论的平行回答。将独立轮、异议挑战与证据合规状态写进协议，至少让不确定性可以被看见和审计。它不保证结论正确，也不应把模型共识误当事实；尤其在 API provider 或证据不足时，仍需要人工审阅其保留的分歧。

## 趋势 / 编辑简评

今天的三个项目分别切入 **学习、运行与协商** 三个层面，却有同一个工程取向：让 Agent 的过程留下可复查的结构。pi-textbook 用 checkpoint 固定学习路径；BossConsole 将工具权限显式化；Co-Agent 将不同模型的反对意见作为结果的一部分。下一步真正值得观察的不是它们能否堆叠更多功能，而是这些结构能否在真实工作流里降低错误成本：能否复现、能否限制、能否解释不同意见为何没有被采纳。
