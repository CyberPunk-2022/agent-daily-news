## 今日要点

1. **Agent 工具开始明确“看见之后何时能行动”。**agent-android 让 UI 元素引用绑定到某次快照；界面变动后旧引用会报错，迫使调用方重新观察。
2. **并行的瓶颈是共享状态，而不只是任务分配。**Beacon 关注谁正在改哪个文件，并把破坏性 Git 操作放进冲突治理范围；它是提示层，不是锁。
3. **记忆系统的价值取决于可追溯性与可操作的限制。**Cortex 同时提供版本、来源和 Merkle 证明，并在基准中说明当前大候选集查询的线性成本。

## 开源项目

### 1. agent-android：面向 Agent 的 Android “看—做—验证”命令行

- **首次公开：**GitHub 仓库创建于 **2026-07-22 00:09 UTC**。
- **项目：**[squaduplabs/agent-android](https://github.com/squaduplabs/agent-android)（MIT）。
- **项目自述：**基于 `adb` 与 `uiautomator` 输出无障碍快照和元素引用，提供点击、拖拽、输入、截图、权限和应用管理等命令；README 说明引用属于单次快照 generation，界面更新后使用旧引用会以错误码提示重新快照。项目不安装设备端 agent 或 instrumentation APK。
- **为什么值得关注：**GUI Agent 常将视觉定位、输入操作和状态确认混成一条难以审计的链路。将每次操作绑定到可读的无障碍树及其代际，能将“界面可能已变”的风险变成可处理的失败条件。无障碍树并不覆盖所有界面，README 也指出安全窗口与持续动画可能需要截图或坐标等降级路径。

### 2. Beacon：给并行 coding agent 一个本地“谁在动什么”的信号面

![Beacon README 展示的并行 Agent 活动面板与文件重叠警告](https://raw.githubusercontent.com/a1473838623/agent-beacon/main/docs/hero.svg)

*图：Beacon 官方 README 使用的活动面板示意图；原始来源：[docs/hero.svg](https://github.com/a1473838623/agent-beacon/blob/main/docs/hero.svg)。*

- **首次公开：**GitHub 仓库创建于 **2026-07-22 03:49 UTC**。
- **项目：**[a1473838623/agent-beacon](https://github.com/a1473838623/agent-beacon)（MIT）。
- **项目自述：**以本机守护进程和活动 `{ actor, action, target }` 模型记录会话正在编辑的目标，提供 report/query、实时面板与冲突提示；README 列出 Claude Code hook、Codex/MCP、脚本和文件监视等接入方式。默认仅监听本机并采用“失败放行”的建议模式；可选 `ask` 模式只在有冲突的破坏性 Git 操作前请求确认。
- **为什么值得关注：**多 Agent 的并发损失往往来自同一文件被覆盖、或一个会话重置了另一个会话的工作树。这里的重点不是替代 Git 或 worktree，而是把隐形冲突提前变成上下文信号。它不提供默认强制锁，且其“零依赖”“零 token”等表述为项目方说明；在团队工作流中仍应以分支、评审与备份作为最终保障。

### 3. Cortex：用版本、来源与状态证明构建可审计的 Agent 记忆

- **首次公开：**GitHub 仓库创建于 **2026-07-22 06:04 UTC**。
- **项目：**[paxlabs-inc/cortex-core](https://github.com/paxlabs-inc/cortex-core)（Apache-2.0）。
- **项目自述：**这是一个嵌入式 Go 库：每个 actor 使用独立 Pebble 数据库，写入类型化、不可变版本的记忆；README 列出结构化/语义/图/时间检索、来源与置信度、可选 AES-256-GCM 加密，以及由日志 Merkle accumulator 和稀疏 Merkle tree 产生的快照与成员证明。项目仍为 pre-1.0，公开 API 可能变化。
- **为什么值得关注：**“把历史放进向量库”难以回答某条上下文从何而来、何时被更新。将来源、版本和证明作为一等模型，给高责任场景提供了更清晰的审计接口。项目还在同日的[基准基线](https://github.com/paxlabs-inc/cortex-core/blob/main/benchmarks/baseline-2026-07-22.md)中披露：当前查询执行器会在应用 `Limit` 前物化、解码和排序候选集，10,000 条 type-only 查询的中位数为 1.89 秒（特定测试环境，非服务承诺）；这类限制应在接入前按目标负载复测。

## 趋势 / 编辑简评

今天的三个项目对应 Agent 的三个基础边界：**界面状态、协作者状态与历史状态**。它们的共同点并不是追求更长的自动执行，而是避免将不确定性藏起来：移动界面更新后让引用失效；多人并行时显式提示重叠；记忆演化时留下版本和证明。下一步值得观察的是这些“可见的边界”能否进入真实工程默认配置——尤其是跨设备 UI、跨工作树协作，以及随数据规模增长的记忆查询。
